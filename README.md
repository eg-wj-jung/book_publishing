# AI 에이전트 팀으로 책 한 권 쓰기

> Claude Code + CLAUDE.md 하나로 176,000자 분량의 책을 처음부터 끝까지 만든 프로젝트입니다.
> 이 레포를 fork해서 CLAUDE.md를 당신의 주제로 바꾸면, 같은 과정을 재현할 수 있습니다.

## 완성물 미리보기

이 프로젝트로 실제 만든 책: **"노트북LM으로 다 됨: 팀장이 바로 쓰는 실전 활용법 20"** (저자: AI ROASTING)

| 항목 | 내용 |
|------|------|
| 분량 | 176,000자, B5 판형 약 240페이지 |
| 구성 | 프롤로그 + 7부(20개 활용법) + 에필로그 + 부록 4개 + 참고문헌 |
| 산출물 | Word 문서(.docx), PDF, 다이어그램 40개 |

최종 산출물은 `output/` 폴더에서 확인할 수 있습니다.

- `output/NotebookLM_final.docx` — 최종 Word 문서
- `output/NotebookLM_final.pdf` — 최종 PDF
- `draft/draft-final.md` — 최종 원고 (Markdown)

## 이 프로젝트가 하는 일

CLAUDE.md 파일 하나에 **책의 주제, 에이전트 팀 구성, 작업 흐름, 품질 기준**을 정의해두면, Claude Code가 이를 읽고 16단계에 걸쳐 책 한 권을 만들어냅니다.

## 핵심 아이디어: 에이전트 팀

혼자 쓰면 자기 글의 약점을 못 봅니다. 이 프로젝트는 **역할이 다른 6명의 에이전트**가 쓰고, 비평하고, 수정하는 구조로 이 문제를 해결합니다.

```
그린(리서처)    →  웹 검색으로 최신 정보 수집
실버(아키텍트)  →  목차와 구조 설계
블랙(라이터)    →  전체 집필 및 수정 (유일한 필자)
레드(리뷰어)    →  냉정한 비평, 논리적 허점 지적
핑크(예상독자)  →  타겟 독자 관점 리뷰
라이트(퍼블리셔) → Word 문서 변환, 서식, 다이어그램
```

최종 퍼블리싱 후에는 **외부 검증자 3명**(편집자, 마케터, 프루프리더)이 10점 만점으로 평가합니다.

### 이 구조가 효과적인 이유

1. **역할 분리**: 쓰는 사람(블랙)과 비평하는 사람(레드)을 분리하면, 자기 글의 약점이 보입니다
2. **다중 관점**: 전문가(레드)와 예상독자(핑크)의 관점이 다르기 때문에, 둘 다 거치면 "전문가에겐 정확하고 초보자에겐 쉬운" 글이 됩니다
3. **반복 개선**: v1 → v2 → v3 → v4 → final로 5번 수정하면서, 매번 다른 관점의 피드백이 반영됩니다
4. **품질 게이트**: 합평에서 9점 미만이면 통과하지 못합니다. 자동으로 품질 하한선이 보장됩니다
5. **외부 검증**: 내부 에이전트끼리의 합의만으로는 부족합니다. 편집자, 프루프리더라는 별도 페르소나가 출판 수준의 검수를 합니다

## 작업 흐름 (16단계)

CLAUDE.md에 이 흐름이 모두 정의되어 있고, Claude Code가 단계별로 실행합니다.

```
[1회전: 초안 만들기]
 1  그린 리서치           → draft/research-notes.md
 2  실버 구조 설계        → draft/outline.md
 3  블랙 초안 작성        → draft/draft-v1.md

[2회전: 비평과 개선]
 4  레드 비평             → draft/review-red.md
 5  블랙 1차 수정         → draft/draft-v2.md

[3회전: 독자 검증]
 6  핑크 독자 리뷰        → draft/review-pink.md
 7  블랙 2차 수정         → draft/draft-v3.md

[합평: 전원 평가]
 8  합평 1라운드 (5명)    → draft/ensemble-review-1.md
 9  블랙 3차 수정         → draft/draft-v4.md
10  합평 2라운드 (5명)    → draft/ensemble-review-2.md
11  블랙 최종 반영        → draft/draft-final.md

[퍼블리싱]
12  라이트 Word 문서 생성 → output/NotebookLM_final.docx

[외부 검증]
13  편집자 평가           → draft/review-editor.md
14  마케터 평가           → draft/review-marketer.md
15  프루프리더 교정       → draft/review-proofreader.md
16  블랙 최종 보강        → draft/draft-final.md (갱신)
```

## 따라하는 법

### 1. 환경 준비

```bash
# 이 레포를 fork 또는 clone
git clone https://github.com/airoasting/notebooklm.git
cd notebooklm

# Python 의존성 설치 (Word 문서 생성용)
pip install python-docx matplotlib

# Claude Code 설치 (Anthropic 공식 CLI)
npm install -g @anthropic-ai/claude-code
```

**폰트 준비**: `font/` 폴더에 Pretendard 폰트 파일이 포함되어 있습니다. 이 폰트가 없으면 Malgun Gothic으로 대체됩니다. macOS 사용자는 Malgun Gothic도 기본 설치되어 있지 않으므로, `font/` 폴더를 유지하거나 원하는 한글 폰트로 교체하세요.

### 2. CLAUDE.md 수정

CLAUDE.md를 열고 **당신의 책에 맞게** 아래 항목을 바꿉니다.

| 바꿀 항목 | 설명 | 예시 |
|-----------|------|------|
| 프로젝트 목표 | 책 제목, 대상 독자, 분량 | "직장인을 위한 ChatGPT 활용법" |
| 블랙의 페르소나 | 필자의 캐릭터, 말투, 필명 | "IT 컨설턴트 출신, 실전파" |
| 최종 산출물 구조 | 프롤로그, 부, 장, 에필로그, 부록 | 원하는 목차 구조로 **통째로 교체** |
| 작성 규칙 | 문체, 금지 표현, 서식 규칙 | em dash 금지, 영문 병기 등 |
| 품질 기준 | 분량, 페르소나 일관성, 필수 요소 | "각 장마다 핵심 정리 박스" |

에이전트 팀 구성(6명)과 작업 흐름(16단계)은 그대로 두어도 됩니다. 이 구조 자체가 품질을 보장하는 장치입니다.

**중요**: "최종 산출물 구조" 섹션은 현재 노트북LM 책의 7부 20개 활용법 목차가 상세히 기술되어 있습니다. 이 부분을 **당신의 책 목차로 통째로 교체**해야 합니다. 부분 수정이 아니라 전체 교체입니다.

### 3. 스크립트 수정

`output/generate_docx.py`와 `output/generate_images.py`는 현재 이 책(노트북LM)에 맞춰 작성되어 있습니다.

| 파일 | 수정할 부분 |
|------|------------|
| `generate_docx.py` | `BASE_DIR` 경로, `OUTPUT_PATH` 파일명, 표지 제목/저자명, 환율 등 |
| `generate_images.py` | 다이어그램 내용 전체 (당신의 책 내용에 맞는 다이어그램으로 교체) |

스크립트를 처음부터 다시 만들 필요는 없습니다. CLAUDE.md에 퍼블리싱 규칙(판형, 폰트, 색상, 서식)이 정의되어 있으므로, Claude Code에게 "12단계 퍼블리싱을 수행해줘"라고 하면 스크립트를 자동으로 생성하거나 수정합니다.

### 4. 실행

```bash
# Claude Code 실행
claude

# 1단계부터 시작하라고 지시
> 1단계부터 시작해줘
```

분량이 많으므로 부(Part) 단위로 끊어서 진행하는 것을 권장합니다.

```
> 3단계 프롤로그와 1부를 작성해줘
> 이어서 2부를 작성해줘
> 4단계 레드 비평을 수행해줘
```

### 5. 주의사항

- **분량 관리**: 12만~15만 자 분량은 한 번에 생성되지 않습니다. 부 단위로 나눠 작성하고 이어붙이는 방식을 씁니다
- **컨텍스트 한계**: 대화가 길어지면 앞부분을 잊습니다. 새 대화를 시작하더라도 CLAUDE.md와 draft 파일을 읽으므로 이어서 작업할 수 있습니다
- **합평 기준**: 합평 2라운드에서 5명 평균 9점 이상이면 최종 퍼블리싱으로 넘어갑니다. 미달이면 재수정 후 합평을 1회 추가합니다 (최대 3라운드)
- **마케터 의견**: 이 프로젝트에서는 마케터의 피드백을 참고만 하고 본문 수정에는 반영하지 않도록 설정했습니다. 당신의 프로젝트에서는 이 규칙을 바꿔도 됩니다

## 예상 소요시간과 비용

이 책(176,000자)을 만드는 데 실제로 걸린 규모감입니다.

| 항목 | 규모 |
|------|------|
| 총 대화 세션 | 약 15~20회 |
| 총 소요시간 | 약 8~12시간 (사람이 모니터링하는 시간 기준) |
| Claude Code 비용 | Claude Pro/Max 구독 또는 API 사용량에 따라 다름 |
| 가장 오래 걸리는 단계 | 3단계(초안 작성) — 부 단위로 7~10회 나눠 진행 |
| 가장 빠른 단계 | 4단계(레드 비평) — 1회 대화로 완료 |

**비용 참고**: Claude Code는 Anthropic API를 사용합니다. Claude Max 구독($100/월 또는 $200/월)을 사용하면 사용량 한도 내에서 추가 비용 없이 진행할 수 있습니다. API 직접 사용 시에는 입출력 토큰에 따라 비용이 발생합니다.

## 트러블슈팅

### 대화 중간에 끊겼을 때

새 대화를 시작하고 이렇게 말하면 됩니다.

```
> draft/draft-v1.md를 읽고, 마지막으로 작성된 부분 이후부터 이어서 작성해줘
```

CLAUDE.md에 "중간에 끊기면 마지막으로 저장된 지점 이후부터 이어서 작성한다"는 규칙이 정의되어 있으므로, Claude Code가 자동으로 이어갑니다.

### 컨텍스트가 너무 길어졌을 때

draft 파일이 길어지면 Claude Code가 전체를 읽지 못할 수 있습니다. 이때는 부 단위로 지시합니다.

```
> draft/draft-v1.md의 제3부만 읽고, 레드 관점에서 비평해줘
```

### 합평에서 9점을 넘기지 못할 때

합평은 최대 3라운드까지 진행됩니다. 3라운드에서도 9점 미만이면 현재 상태에서 퍼블리싱을 진행하고, 외부 검증 단계에서 추가 보강합니다.

### 다른 언어로 책을 쓰고 싶을 때

CLAUDE.md의 작성 규칙에서 언어와 문체를 바꾸면 됩니다. 에이전트 이름이나 작업 흐름은 언어와 무관하게 동작합니다. 단, `generate_docx.py`의 폰트 설정은 해당 언어에 맞게 수정해야 합니다.

### Word 문서 생성 시 폰트가 깨질 때

`font/` 폴더에 Pretendard 폰트가 있는지 확인하세요. 없으면 Malgun Gothic으로 대체되는데, macOS에서는 이 폰트도 없을 수 있습니다. 이 경우 `generate_docx.py`의 `FONT_FALLBACK`을 시스템에 설치된 한글 폰트(예: `AppleGothic`, `NanumGothic`)로 변경하세요.

## docx 생성기 주요 기능

`generate_docx.py`가 지원하는 서식 기능:

| 기능 | 설명 |
|------|------|
| 표 페이지 분할 방지 | 10행 이하 표는 `keepTogether`로 한 페이지 유지, 긴 표는 분할 허용 |
| 이미지-캡션 동일 페이지 | `keepNext`로 이미지와 캡션이 항상 같은 페이지에 표시 |
| 셀 내 줄바꿈+작은 글씨 | `<br><sm>텍스트</sm>` 태그로 줄바꿈 후 8pt 글씨 처리 |
| 핵심 정리 박스 | 배경색 #E8F0FE, 테두리 #1F4E79, `keepTogether` 적용 |
| 자동 이미지 삽입 | `[그림 N: 설명]` 패턴 감지 → `output/images/figNN.png` 삽입 |
